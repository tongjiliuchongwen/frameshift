# v0.4 — Interactive human-in-the-loop via the reframe-ui channel

## Why
Through v0.3.1 the workflow was already traceable and human-gated, but the human interaction
"bottomed out" at: read a static `04_method_grid.html`, then type the selection back into the
Claude Code / Codex conversation. That is correct but semi-manual. v0.4 closes that loop with a
clickable UI whose selections are delivered straight into the running session — without turning the
project into a full web app or adding a heavy dependency.

## What it is (not)
- It **is** an optional Claude Code *channel*: a localhost MCP plugin (`reframe-ui/`) that pushes the
  user's gate selection into the already-open session as a `<channel source="reframe-ui" …>` event.
- It **is not** a replacement for the chat-mediated gates (those still work and remain the default),
  not a hosted web app, and not a Codex feature (Codex has no channel; its equivalent path is the
  App Server / `codex exec`).

## Architecture
One Node process = (a) MCP server over stdio (the channel) + (b) HTTP/SSE server on `127.0.0.1`.

```
browser UI  --POST /select-->  reframe-ui process  --notifications/claude/channel-->  Claude session
browser UI  <--SSE /events---  reframe-ui process  <--reply tool------------------ --  Claude session
```

- Becomes a channel by declaring `capabilities.experimental["claude/channel"] = {}`.
- Inbound: `mcp.notification("notifications/claude/channel", { content, meta })` → arrives as a
  `<channel …>` tag whose body (`content`) is a directive and whose attributes (`meta`: gate, mode,
  run_id, cells/candidates) are routing. `meta` holds ids/enums only; free text goes in `content`.
- Outbound: a standard `reply` MCP tool the agent calls to push an SSE status so the UI advances.
- The browser side reads run state from `GET /state` (which reads the run's `0x_*.json` from disk) and
  branches on `gate` ∈ {1, 2, done, not_ready}.

See `reframe-ui/CONTRACT.md` for the exact wire shapes and `reframe-ui/README.md` to run it.

## Design choices
- **Node + SSE, not Bun + WebSocket.** The channel contract only needs `@modelcontextprotocol/sdk`
  plus a Node-compatible runtime; SSE drops the WebSocket dependency. (The official sample channels
  use Bun, which is not required.)
- **stdout is reserved for MCP.** All logging goes to stderr; emitting to stdout would corrupt the
  JSON-RPC transport.
- **State from disk, not in-memory.** The UI reflects whatever the pipeline has written, so it works
  with the existing artifacts unchanged and survives reloads.
- **Three modes, one contract.** 精确 / 人话 / 托管 all reduce to a single `POST /select` body; the
  agent maps natural-language and delegated intent to ids.

## Gate flow
1. `/reframe-workshop` runs to Gate 1 → `03_method_grid.json` exists → UI shows cell selection.
2. User picks cells → channel event → agent runs `method-qd-search` on them → `04_*` → UI → Gate 2.
3. User picks candidates → channel event → agent runs `idea-card` + validator → `06_*` → UI → done.

## Limitations
Research preview (the `claude/channel` contract may change); Claude Code v2.1.80+; Anthropic auth
(not on Bedrock/Vertex/Foundry); the session must stay open; custom channels require
`--dangerously-load-development-channels`. Pro/Max personal accounts work by default; Team/Enterprise
must enable `channelsEnabled`.

## Verification done at build time
`node --check` + a 7-check `smoke-test.mjs` (server boot, `/runs`, `/state` shape, `/select`
validation + happy path, SSE hello) all pass; the gate-1 → gate-2 → done UI states were each rendered
and screenshotted against real runs; the `/select` handler was confirmed to emit a correctly-shaped
`notifications/claude/channel`. The only link not exercisable from a non-channel session — the event
actually landing *in a session* — is covered by the user running `--dangerously-load-development-channels`.
