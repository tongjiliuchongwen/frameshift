# v0.5 — Three-gate redesign: System → Lateral → Vertical

Status: **DESIGN / IN PROGRESS** (branch `v0.5-three-gate`). This is a contract-breaking major
revision. It retires the v0.2 `lateral-reframe` linear path and the v0.3 user-facing *method-grid
gate*, and reshapes the engine around the original systems / lateral / vertical cognitive split.

## Why

Through v0.4.1 the human interacted with the pipeline at two gates:

```
Gate 1: pick a method cell  (system_leverage_type × lateral_operator)
Gate 2: pick a QD candidate → becomes an idea card
```

Two problems:
1. **Gate 1 is abstract.** It asks the human to choose a *method combination*
   (`feedback_information × reversal`) instead of a *place in the system* (`LP-07: the
   trace/evaluation feedback loop`). The human cares about where to cut, not which lateral trick.
2. **Generation and judgment are fused.** QD candidates are scored on a 5-dim heuristic *at
   generation time*, so a lateral idea is appraised the instant it appears. De Bono's whole point is
   the opposite: lateral thinking *defers* judgment to stay generative; vertical thinking *then*
   selects and deepens. The v0.4 pipeline never has a real "stay open" phase, and never has a real
   audit phase either — the 5-dim score is neither.

## New pipeline

```
01 system_map
02 leverage_points
  ── Gate 1: select LEVERAGE POINTS (where in the system to cut) ──
03 lateral_reframes        (diverge; operators internal; JUDGMENT DEFERRED)
  ── Gate 2: select LATERAL SCHEMES (which divergent moves are interesting) ──
04 vertical_audits         (converge; ADVERSARIAL default-reject; external auditor)
  ── Gate 3: select AUDITED SCHEMES (which survive and are worth a card) ──
06 idea_cards
```

Three gates map to three distinct kinds of human judgment:

| Gate | Object | Cognitive mode | Human question |
|------|--------|----------------|----------------|
| 1 | leverage points (`LP-###`) | systems thinking | "Where in this system is it worth cutting?" |
| 2 | lateral schemes (`LR-###`) | lateral thinking | "Which divergent moves are *interesting* enough to audit?" (protect the unproven) |
| 3 | audited schemes (`VA-###`) | vertical thinking | "Which *survive scrutiny* and deserve a card?" |

Gate 2 and Gate 3 are deliberately separate: Gate 2 protects "interesting but unproven", Gate 3
keeps "still standing after audit". Collapsing them (today's single candidate→card gate) is the
root of the dissatisfaction.

## Trace chain

```
input_evidence → system_node → LP-### → LR-### → VA-### → IC-###
```

Replaces the v0.3 chain (`… → CELL-### → QD-### → IC-###`). The idea-card `method_trace`
byte-equality contract is re-pointed from "QD candidate" to "audited scheme (VA)", which itself
carries its source lateral scheme (LR) and leverage point (LP).

## Stage specs

### Stage: lateral-generate → `03_lateral_reframes.{json,md,html}`
- Input: `02_leverage_points.json` + the Gate-1 selection.
- For each selected `LP-###`, internally sweep multiple lateral **operators** (the 8-operator set is
  an *internal generation mechanism, not a user-facing axis*). The grid lives on here as an internal
  **coverage ledger** keyed by `LP × operator` (see "Grid demotion").
- **Judgment is deferred.** A lateral scheme carries NO quality score. It carries
  `not_yet_audited: true`, the operator that produced it, `old_frame`/`lateral_move`/`new_frame`,
  the scheme, and `why_interesting`.
- **Anti-collapse guardrails are kept** (this is the project's hardest-won asset — the sibling
  `frameshift` project spent three rounds fixing diversity collapse; here the operator partition +
  duplicate-value checks prevent it). Each scheme must have a distinct `new_frame` /
  `changed_assumption`, carry an operator-specific `bad_use_to_avoid`, and the validator keeps the
  duplicate-value WARNs (hardened against CJK / ID-based grounding, see validator notes).

`LR` shape (abridged):
```json
{
  "lateral_id": "LR-001",
  "source_leverage_point": "LP-07",
  "operator": "reversal",
  "old_frame": "...",
  "lateral_move": "...",
  "new_frame": "...",
  "scheme": "...",
  "why_interesting": "...",
  "changed_assumption": "...",
  "bad_use_to_avoid": "...",
  "not_yet_audited": true
}
```

### Stage: vertical-audit → `04_vertical_audits.{json,md,html}`
- Input: Gate-2-selected `LR-###` schemes.
- **Adversarial, default-reject.** Each scheme is presumed `reject` and survives only by passing
  three gates: (a) a concrete *falsifiable minimal experiment* is describable; (b) it is
  *discriminable* from obvious prior work (not a renamed known idea); (c) it answers *so-what*.
- **Dual adversarial judges (Codex + Claude).** Each Gate-2-selected scheme is audited
  INDEPENDENTLY by both an external engine (Codex via `codex exec`, read-only sandbox, schema-forced
  verdict — NOT the model that generated the scheme) and Claude, each under the same default-reject
  rubric. **Agree → take the agreed verdict. Disagree → escalate** (surface both verdicts to the
  human at Gate 3, or re-audit). Each `VA` records `codex_verdict`, `claude_verdict`, the resolved
  `verdict`, and `auditor ∈ {dual, codex, self}` (degrades to a single judge with honest provenance
  if Codex is unavailable).
- Output per scheme: verdict (`reject` / `revise` / `keep`), the three gate booleans, refined scheme
  (on `revise`), core_claim, causal_mechanism, critical_assumptions, novelty_risk, testability,
  minimal_experiment, failure_modes, merge_suggestions, and an `audit_score` block. Unlike the v0.3
  archive, the `reject` verdict is expected to **actually fire** and is preserved with reasons (the
  "rejected drawer" that v0.3 never filled).

`VA` shape (abridged):
```json
{
  "audit_id": "VA-001",
  "source_lateral_id": "LR-001",
  "auditor": "dual",             // dual | codex | self
  "codex_verdict": "reject",     // reject | revise | keep
  "claude_verdict": "revise",
  "agreement": false,
  "verdict": "revise",           // resolved; on disagreement → escalated to Gate 3
  "minimal_experiment_exists": true,
  "discriminable_from_prior": true,
  "so_what_passes": true,
  "refined_scheme": "...",
  "core_claim": "...",
  "causal_mechanism": ["..."],
  "critical_assumptions": ["..."],
  "novelty_risk": "...",
  "minimal_experiment": "...",
  "failure_modes": ["..."],
  "merge_suggestions": [{ "with": "LR-004", "reason": "..." }],
  "audit_score": { "coherence": 4, "testability": 5, "novelty_potential": 4, "research_value": 5, "risk": 3, "overall": 4.4 },
  "reasons": ["..."]
}
```

### Stage: idea-card → `06_idea_cards.{json,md}`
- Input: Gate-3-selected `VA-###` schemes (NOT raw lateral ideas).
- `method_trace` now references the audited scheme; the card inherits the audit's minimal experiment
  / failure case rather than re-inventing them.

## Grid demotion (method-grid / QD-lite)

The `6 system_leverage_type × 8 lateral_operator` grid is **no longer a user-facing gate**. It is
retained as an **internal coverage ledger** inside lateral-generate: for each selected `LP`, the
ledger records which operators were tried and how many schemes survived dedup. This keeps the
anti-collapse property and the "empty space is information" inspection signal, without forcing the
human to choose method cells. `candidate_quota` (assigned but never consumed in v0.4) is either given
real meaning by the ledger or removed.

## Canonical contract module (P0 — single source of truth)

Today the vocabulary/version constants are duplicated across ~6 locations (validator,
`render_method_grid.py`, three schemas, the UI contract) with the operator set additionally **split**
between v0.2 and v0.3. v0.5 introduces ONE canonical module that the schemas' enums, the validator's
constants, the renderer lists, and the UI contract are all generated from / checked against:

- **lateral_operators** — a single unified set of 8 (resolving the v0.2/v0.3 split). v0.5 adopts the
  v0.3 set `{assumption_challenge, reversal, decomposition, analogy, random_stimulus, PO_provocation,
  entry_point_shift, concept_abstraction}` and **drops** the v0.2-only
  `{function_abstraction, boundary_shift, goal_inversion}`. The idea-card validator's permissive
  *union* check is replaced by an exact enum.
- **leverage_types** — leverage-scan's 10 Meadows-style types stay as *metadata on each LP*, no
  longer a grid row axis.
- **verdict** — `{reject, revise, keep}`.
- **id patterns** — widened to 3 digits to remove the v0.4 scaling ceiling:
  `LP-###`, `LR-###`, `VA-###`, `IC-###` (was `LP-##` / `IC-##`, capped at 99).
- **schema_version** — bumped from `"1.0"` to `"2.0"` (contract-breaking).
- **artifact filenames + trace chain** — defined once.

## Validator changes (P2)

- Rewrite cross-artifact trace integrity for `LP → LR → VA → IC` (byte-equality re-pointed to VA).
- Coverage arithmetic moves from the 48-cell grid to the internal `LP × operator` ledger.
- **Close the audit-trail holes**: validate that `05_human_selection.md` / `decision_log.md` exist
  and are consistent (today nothing checks them, so "two/three gates" is author-discipline only).
- Harden the `why_not_obvious` specificity heuristic against CJK and ID-based grounding (it currently
  false-flags grounded candidates via an ASCII-only tokenizer).
- Keep the dual root/bundled copies byte-identical via `check_bundle_consistency.py` (extend it to
  cover `render_method_grid.py` and the UI contract).

## reframe-ui changes (P3)

- Add a **third gate state** (`gate ∈ {1, 2, 3, done, not_ready}`); today it is `{1, 2, done}`.
- Gate detection (`buildState()`) currently keys on raw file existence and never runs the validator,
  so it can report `gate=done` on a structurally-invalid artifact. v0.5 has the agent run the
  validator before each `reply`, or makes the gate guard depend on contract validity.
- Gate 1 now selects leverage points; Gate 2 selects lateral schemes; Gate 3 selects audited schemes.
  The `ungrounded_empty` status (in schema/validator but absent from `ui.html`) is rendered.

## Codex external-audit integration (validated)

Proof-of-concept run 2026-06-27: `codex-cli 0.139.0`, authenticated via `~/.codex/auth.json`.

```
codex exec --skip-git-repo-check -s read-only \
  --output-schema verdict.schema.json -o verdict_out.json - < audit_prompt.txt
```

Returned a clean schema-shaped verdict in ~41s / ~15k tokens, and was genuinely adversarial
(rejected a deliberately weak scheme, identifying it as reflection/self-critique memory). Read-only
sandbox is sufficient — the auditor only reads. **Dual judging**: each Gate-2 scheme also gets
Claude's own adversarial pass; agreement is taken directly, disagreement is escalated to the human
at Gate 3 rather than silently resolved. Per-run cost for a 4–6 scheme batch is a handful of
`codex exec` calls plus Claude's pass. Honest provenance: every `VA` records both judges' verdicts
and `auditor ∈ {dual, codex, self}`.

## Deletions (retired in v0.5)

- `skills/lateral-reframe/` (v0.2 legacy path) — its *concept* is promoted into lateral-generate; its
  old wiring is removed.
- `schemas/reframes.schema.json` + the bundled copy; the validator's v0.2 layout branch and `RF-##`
  detection; idea-card's v0.2 legacy branch.
- The three thin v0.2 examples (`weak-ai-research-idea`, `paper-gap-reframe`,
  `open-source-roadmap-reframe`) — replaced by 1–2 real non-meta v0.5 end-to-end examples.
- The v0.2-only operators (`function_abstraction`, `boundary_shift`, `goal_inversion`).

**NOT a deletion (corrected 2026-06-27):** an earlier audit flagged the reframe-ui `node_modules` /
lockfile (~93 packages: express, hono, cors, zod, jose, ajv…) as "phantom deps left over from an
abandoned Express approach." That was a **misdiagnosis**. `npm ls` confirms every one of those is a
declared transitive dependency of `@modelcontextprotocol/sdk@1.29.0` (it bundles HTTP/SSE transports,
CORS, OAuth via jose/pkce-challenge, and zod schema validation). A clean install of the single
declared dependency reproduces the exact tree. There is nothing to delete here; the heaviness is
inherent to the official MCP SDK and could only shrink by vendoring/replacing it (out of scope for
v0.5).

## Build sequence & regression gates

- **P0** canonical contract module + `schema_version` bump (land while green).
- **P1** lateral-generate + vertical-audit skills; reframe-workshop re-orchestration.
- **P2** validator rewrite for the new chain + audit-trail checks.
- **P3** reframe-ui third gate + validity-based gate detection; new examples; static renderer.

After every step the regression gate is: `validate_outputs.py` exits 0 on the current examples and
`check_bundle_consistency.py` passes. These are green today and must stay green.

## Backward compatibility

None preserved by choice. v0.5 is a clean break: artifacts written by v0.2/v0.3 will not validate
under the v2.0 contract. `main` (tag `v0.4.1`) remains the last v0.4 release for anyone needing it.
