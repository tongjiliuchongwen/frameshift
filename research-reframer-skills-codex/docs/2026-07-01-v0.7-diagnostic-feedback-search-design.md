# v0.7 Design: Diagnostic Feedback Search

Status: IMPLEMENTED BASELINE

v0.7 keeps the v0.5 three-gate workflow and adds the missing feedback-search pieces around it. The goal is not to turn the project into a longer methodology pipeline. The goal is to make every generated idea traceable to an initial diagnosis, grounded in questionable-but-auditable system relations, filtered by explicit pseudo-innovation failure modes, and reported as a clear change from the original idea.

## Current Baseline

The current v0.5.1 flow is:

```text
input
-> system-map
-> leverage-scan
-> Gate 1: choose leverage points
-> lateral-generate
-> Gate 2: choose lateral schemes
-> vertical-audit
-> Gate 3: choose audited schemes
-> idea-card
```

This is already better than the linear "system -> lateral -> vertical -> output" model. It separates generation from judgment and records human gate choices. v0.7 should preserve that split.

## Problems v0.7 Should Fix

1. There is no Stage 0 diagnosis, so the system starts mapping before it names what is unsatisfactory about the original idea.
2. The system map has structured nodes, but key relations do not carry relation type, evidence, confidence, or "what breaks if this edge is wrong".
3. The global quality target is implicit. Different skills do not share a concise contract for what counts as a good research idea.
4. Pseudo-innovation is detected informally by prompts and vertical-audit reasons, but failure types are not machine-readable.
5. Vertical audit currently mixes `reject` with `needs_human` rescue semantics, which makes examples read as if an idea passed the gates but was still rejected.
6. Final idea cards have trace fields, but not a compact change log explaining how the final idea differs from the original.
7. Human taste feedback, failure memory, and strategy weights are not yet cross-run learning state.

## Non-Goals

- Do not add a hard pseudo-innovation gate before Gate 2. That would collapse v0.5's intended separation between lateral generation and vertical judgment.
- Do not replace the canonical eight lateral operators. If v0.7 needs "change variable / relation / boundary / goal" labels, add them as structural-change metadata or self-check dimensions.
- Do not make failure memory or strategy weights block the first v0.7 release. They are useful, but they add cross-run state and should come after the contract is stable.
- Do not change the sidecar-only gate interaction model.

## Proposed v0.7 Flow

```text
input
-> 00_diagnosis
-> 01_system_map
-> 02_leverage_points
-> Gate 1: choose leverage points
-> 03_lateral_reframes
-> Gate 2: choose lateral schemes
-> 04_vertical_audits
-> Gate 3: choose audited schemes
-> 06_idea_cards
-> report + decision log
```

The numbering leaves existing artifact names mostly intact while inserting `00_diagnosis.json` before the current chain.

## P0 Contract Changes

### 1. Global Quality Contract

Add a shared quality contract to `reframe-workshop` and echo it in the child skills:

```text
The final objective is a research idea that has a structural increment over the original idea,
a plausible scientific mechanism, a minimal verification path, and a plain-language explanation
of why the idea is genuinely different rather than a terminology swap.
```

This is a prompt-level change first. It does not need a schema field.

### 2. `00_diagnosis.json`

Add a new artifact before `01_system_map.json`.

Required fields:

```json
{
  "schema_version": "2.1",
  "original_idea_summary": "...",
  "dissatisfaction_types": ["novelty_gap", "unclear_mechanism"],
  "primary_bottleneck": "...",
  "core_value_to_preserve": "...",
  "most_promising_change_target": "...",
  "downstream_focus": "..."
}
```

Allowed dissatisfaction types:

```text
novelty_gap
unclear_mechanism
weak_testability
jargon_or_opacity
loose_structure
unclear_prior_difference
too_broad_to_execute
too_conservative
```

Downstream artifacts should be able to state which diagnosis bottleneck they address, but v0.7 MVP can enforce this in reports and validator warnings before making it a hard error.

### 3. Questionable System Relations

Extend key relation objects in `system_map` with relation audit metadata.

For `flows`, `feedback_loops`, and `information_flows`, add:

```json
{
  "relation_type": "fact|inference|analogy|low_confidence",
  "evidence_source": "...",
  "mechanism": "...",
  "if_wrong_impact": "..."
}
```

The validator should require these fields for the relation arrays that already exist. This turns system maps from polished diagrams into auditable claims.

### 4. Pseudo-Innovation Block in Vertical Audit

Do not create a separate pre-Gate-2 auditor. Instead, add a machine-readable block inside each vertical audit:

```json
{
  "pseudo_innovation": {
    "status": "clear|repairable|fatal",
    "failure_types": ["terminology_swap", "glue_splice"],
    "evidence": "...",
    "repair": "..."
  }
}
```

Failure types:

```text
terminology_swap
glue_splice
grand_narrative
template_fill
jargon_masking
unverifiable_divergence
over_conservative_repair
```

The vertical audit can still output `reject`, `revise`, or `keep`; the new block explains why.

### 5. Cleaner Escalation Semantics

Current examples use `verdict: reject` with `needs_human: true` when judges disagree. v0.7 should make that state easier to read.

Recommended minimal change:

- keep `verdict` as the conservative machine verdict for backward compatibility;
- add `escalation_reason`;
- add `human_resolution` after Gate 3 when the human rescues an audit.

This avoids pretending the machine "kept" an idea while still making human rescue explicit.

### 6. Idea Change Log

Add a required `change_log` block to each idea card:

```json
{
  "change_log": {
    "source_diagnosis": "primary bottleneck addressed",
    "selected_leverage_point": "LP-###",
    "structural_change": "...",
    "new_mechanism": "...",
    "preserved_from_original": "...",
    "removed_or_weakened": "...",
    "real_increment_over_original": "...",
    "remaining_fragility": "..."
  }
}
```

This makes the final artifact answer: where it came from, what changed, why it is better, and how it can still fail.

## P1 / Later

### Human Taste Feedback

Extend human selection only after the sidecar UX is ready. A future `taste_feedback` block can record:

```text
liked candidates
disliked candidates
jargon complaints
glue complaints
true novelty signals
rigorous-but-boring signals
next-round strengthen
next-round avoid
```

### Failure Memory

Add `failure_memory.json` only after pseudo-innovation failure types are stable. It should start as an optional run artifact, not a global mandatory state file.

### Strategy Weights

Defer `strategy_weights.json`. It touches operator selection, coverage arithmetic, and exploration/exploitation behavior, so it should not be bundled into the first v0.7 cut.

## Implementation Order

The approved implementation proceeded in this order:

1. Update contract/schema/validator for `00_diagnosis`, relation audit fields, `pseudo_innovation`, and `change_log`.
2. Update bundled references and sync checks.
3. Update skill prompts to generate the new fields.
4. Regenerate or patch two complete example runs.
5. Update README/CHANGELOG and run validation.
